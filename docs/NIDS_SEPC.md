# NIDS报文处理管道抽象设计提示词（优化版）

## 角色与目标
你是一位资深网络安全架构师，专精于高性能NIDS（网络入侵检测系统）设计。
你需要为NIDS的报文处理管道设计一个高性能、可扩展的抽象实现方案，该方案需要处理高吞吐量网络流量（至少10Gbps），并支持灵活的检测规则扩展。

## 输入上下文

### 管道阶段定义
pipline 由多个stage/mod 组成:

1. 抓包
2. 报文预处理
3. 报文解码
4. 报文检测引擎
5. 报文匹配引擎
6. 事件生成引擎

*说明：这些阶段定义了报文从捕获到处理完成的全流程*

### 线程配置要求
- 主线程
    - NIDS　应用配置，启动,生命周期管理

- 监控线程
    - 监控CPU, MEM ,按照 阈值调整抓包功能开启/停止

- 通信线程
    - 通过IPC 接受外部控制命令,发送SecEvent,接受动态规则下发

- 每个网卡对应一个pipline, 每个pipline 包含
    - 抓包线程
        - Packet 内存池相关初始化
        - 抓包
    - 报文处理解析匹配线程
        - 预处理
        - 解码
        - 检测
        - 匹配
        - 事件生成

*说明：定义了抓包线程与工作线程的数量、CPU亲和性等配置*

## 设计约束与要求

### 1. 架构原则
- **单一职责**：每个管道阶段职责清晰，独立可测
- **松耦合**：各阶段通过统一接口交互，便于替换和扩展
- **高性能**：零拷贝优化，最小化锁竞争，支持NUMA感知
- **线程安全**：多线程环境下数据一致性保证

### 2. 技术规范
- **语言标准**：C++17或更高版本（优先使用std::shared_ptr, std::move等现代特性）
- **并发模型**：生产者-消费者模式，无锁队列优先
- **内存管理**：对象池/内存池技术，避免频繁分配释放
    - PacketPool
        - PakcetSlot
    - PacketPtrQueue
    > PacketPool 是存放PacketSlot 的容器, 预分配内存 +　支持有限的扩展
    > PacketSlot 是真正的存放Packet 的容器，　多个slot size(256B, 512B, 1204B, 2048B)
    > PacketPtrQueue 是存放PacketSlot 的指针

- **性能目标**：单核处理能力≥100,000 pps（packets per second）

- **功能要求**:
    - 抓包
    - 报文处理&分析
    - DDOS 检测
    - snort3 的规则解析,匹配
    - SecEvent 聚合过滤,生成落盘

### 3. 核心功能需求
- **阶段抽象**：定义统一的Stage接口，支持链式调用
- **管道编排**：动态配置管道流程，支持条件分支和并行处理
- **线程模型**：抓包线程与处理线程解耦，支持多队列和流量分发
- **可观测性**：内置性能监控点，支持统计每阶段处理延迟和吞吐量
- **错误处理**：异常报文隔离机制，确保单个报文错误不影响整体流程
